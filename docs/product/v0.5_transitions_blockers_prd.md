## 💾 1. Database Schema & SQL DDL Migration

To support card-level blocker details, isolated blocker discussions, and immutable event auditing, the following PostgreSQL DDL migration must be implemented in: `apps/api/migrations/20260530000000_transitions_blockers.sql`.

```sql
-- Migration: Add Card Blocker Fields and Blocker Discussion Comments
-- Target File: apps/api/migrations/20260530000000_transitions_blockers.sql

-- 1. Add blocker-related columns to cards table
ALTER TABLE cards ADD COLUMN IF NOT EXISTS is_blocked BOOLEAN NOT NULL DEFAULT FALSE;
ALTER TABLE cards ADD COLUMN IF NOT EXISTS blocked_by UUID REFERENCES users(id) ON DELETE SET NULL;
ALTER TABLE cards ADD COLUMN IF NOT EXISTS blocked_at TIMESTAMPTZ;
ALTER TABLE cards ADD COLUMN IF NOT EXISTS blocked_reason VARCHAR(255);

-- 2. Create card_block_comments table for blocker-isolated discussion threads
CREATE TABLE IF NOT EXISTS card_block_comments (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    card_id UUID NOT NULL REFERENCES cards(id) ON DELETE CASCADE,
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    content VARCHAR(500) NOT NULL CHECK (char_length(trim(content)) >= 1),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Optimize comment retrievals for blocked cards ordered chronologically
CREATE INDEX IF NOT EXISTS idx_card_block_comments_card_id ON card_block_comments(card_id, created_at ASC);

-- 3. Integrity Constraint check for Blocker parameters consistency
ALTER TABLE cards DROP CONSTRAINT IF EXISTS chk_card_blocked_state_integrity;
ALTER TABLE cards ADD CONSTRAINT chk_card_blocked_state_integrity CHECK (
    (is_blocked = FALSE AND blocked_by IS NULL AND blocked_at IS NULL AND blocked_reason IS NULL) OR
    (is_blocked = TRUE AND blocked_by IS NOT NULL AND blocked_at IS NOT NULL AND blocked_reason IS NOT NULL AND char_length(trim(blocked_reason)) >= 1)
);
```

### 2.2 Concurrency-Safe SQL Transactional Workflows

#### A. Card Block Toggle Routine (`Card::block_card`)
When a card is blocked, the service layer MUST lock the row pessimistically to serialize concurrent toggles and prevent race conditions.

```rust
impl Card {
    /// Pessimistically locks a card and declares it blocked within a database transaction.
    pub async fn block_card(
        pool: &sqlx::PgPool,
        workspace_id: Uuid,
        actor_id: Uuid,
        card_id: Uuid,
        reason: String,
    ) -> Result<Self, crate::AppError> {
        let mut tx = pool.begin().await?;

        // 1. Pessimistic Write Lock card row and verify workspace boundaries
        let card = sqlx::query_as::<_, Card>(
            "SELECT * FROM cards WHERE id = $1 AND workspace_id = $2 FOR UPDATE"
        )
        .bind(card_id)
        .bind(workspace_id)
        .fetch_one(&mut *tx)
        .await
        .map_err(|e| match e {
            sqlx::Error::RowNotFound => crate::AppError::NotFound,
            _ => crate::AppError::Database(e),
        })?;

        // Ensure card is not already blocked
        if card.is_blocked {
            return Err(crate::AppError::BadRequest("Card is already blocked".into()));
        }

        // 2. Perform the update in cards table
        let blocked_at = Utc::now();
        let updated_card = sqlx::query_as::<_, Card>(
            r#"
            UPDATE cards
            SET is_blocked = TRUE,
                blocked_by = $1,
                blocked_at = $2,
                blocked_reason = $3,
                updated_at = NOW()
            WHERE id = $4 AND workspace_id = $5
            RETURNING *
            "#
        )
        .bind(actor_id)
        .bind(blocked_at)
        .bind(&reason)
        .bind(card_id)
        .bind(workspace_id)
        .fetch_one(&mut *tx)
        .await?;

        // 3. Write immutable event audit trail record to card_transitions
        sqlx::query(
            r#"
            INSERT INTO card_transitions (
                card_id, user_id, transition_type, payload
            )
            VALUES ($1, $2, 'card_blocked', $3)
            "#
        )
        .bind(card_id)
        .bind(actor_id)
        .bind(serde_json::json!({
            "blocked_by": actor_id,
            "blocked_at": blocked_at,
            "reason": reason
        }))
        .execute(&mut *tx)
        .await?;

        tx.commit().await?;
        Ok(updated_card)
    }
}
```

#### B. Card Unblock Toggle Routine (`Card::unblock_card`)
Resolving a blocker executes an identical transactional flow, clearing card blocker fields and logging the resolution details.

```rust
impl Card {
    /// Pessimistically locks a card and resolves its block status within a database transaction.
    pub async fn unblock_card(
        pool: &sqlx::PgPool,
        workspace_id: Uuid,
        actor_id: Uuid,
        card_id: Uuid,
        resolution: String,
    ) -> Result<Self, crate::AppError> {
        let mut tx = pool.begin().await?;

        // 1. Pessimistic Write Lock card row and verify workspace boundaries
        let card = sqlx::query_as::<_, Card>(
            "SELECT * FROM cards WHERE id = $1 AND workspace_id = $2 FOR UPDATE"
        )
        .bind(card_id)
        .bind(workspace_id)
        .fetch_one(&mut *tx)
        .await
        .map_err(|e| match e {
            sqlx::Error::RowNotFound => crate::AppError::NotFound,
            _ => crate::AppError::Database(e),
        })?;

        // Ensure card is actively blocked
        if !card.is_blocked {
            return Err(crate::AppError::BadRequest("Card is not currently blocked".into()));
        }

        let blocked_duration_seconds = Utc::now()
            .signed_duration_since(card.blocked_at.unwrap_or(card.created_at))
            .num_seconds();

        // 2. Clear blocker fields in cards table
        let updated_card = sqlx::query_as::<_, Card>(
            r#"
            UPDATE cards
            SET is_blocked = FALSE,
                blocked_by = NULL,
                blocked_at = NULL,
                blocked_reason = NULL,
                updated_at = NOW()
            WHERE id = $1 AND workspace_id = $2
            RETURNING *
            "#
        )
        .bind(card_id)
        .bind(workspace_id)
        .fetch_one(&mut *tx)
        .await?;

        // 3. Write immutable event audit trail record to card_transitions
        sqlx::query(
            r#"
            INSERT INTO card_transitions (
                card_id, user_id, transition_type, payload
            )
            VALUES ($1, $2, 'card_unblocked', $3)
            "#
        )
        .bind(card_id)
        .bind(actor_id)
        .bind(serde_json::json!({
            "unblocked_by": actor_id,
            "resolution": resolution,
            "blocked_duration_seconds": blocked_duration_seconds
        }))
        .execute(&mut *tx)
        .await?;

        tx.commit().await?;
        Ok(updated_card)
    }
}
```

### 2.3 Transition Interception & Rejection Policies

When a card is moved between columns or swimlanes, the backend `Card::move_to` service method executes standard active validations. The flow checks for blocker states, capacity WIP limit boundaries, and visual arrival/departure checklists:

```
                            Card Move Transition Pipeline

       User Action         1. Block State        2. WIP Limit Check      3. Transition Rules
    [PUT .../move]   ==>   [Card Blocked?]   ==>   [Target Limits]   ==>   [Arrival/Departure]
                                 ||                      ||                    ||
                            YES: Reject             YES: Reject           YES: Reject
                            (422 Blocked)          (409 Conflict)        (422 Violation)
```

1. **Rule 1: Blocked Card Movement Prevention**:
   - Policy: Blocked cards **CANNOT** be moved between columns or swimlanes.
   - Validation Check: If `card.is_blocked` evaluates to `true`, instantly reject the execution.
   - Rejection Status: `422 Unprocessable Entity`
   - JSON Response Payload:
     ```json
     {
       "error": "CARD_IS_BLOCKED",
       "message": "Blocked cards cannot be moved between columns or swimlanes. Resolve the block details first."
     }
     ```

2. **Rule 2: WIP Capacity Limits**:
   - Policy: Active columns, swimlanes, or assigned users cannot exceed defined threshold limits.
   - Rejection Status: `409 Conflict`
   - JSON Response Payload:
     ```json
     {
       "error": "WIP_LIMIT_EXCEEDED",
       "message": "Move rejected because the target column's Work-in-Progress limit would be exceeded.",
       "entity": "column",
       "limit": 5
     }
     ```

3. **Rule 3: Column Transition Policies (Arrival & Departure Checklists)**:
   - Policy: Cards entering or leaving columns must fulfill active transition rules criteria.
   - Rejection Status: `422 Unprocessable Entity`
   - JSON Response Payload:
     ```json
     {
       "error": "RULE_VIOLATION",
       "message": "[Arrival Rule Violation]: QA requires an Assignee before entry."
     }
     ```

### 2.4 REST API Route Specifications

All endpoints are protected under the workspace context using the `WorkspaceTenantGuard` middleware. This guarantees secure multi-tenant isolation, ensuring users can only read or modify cards in workspaces where they hold active membership.

#### A. Card Block Toggle Endpoint
*   **Endpoint**: `POST /api/workspaces/{workspace_id}/cards/{card_id}/block`
*   **Access Control**: Authenticated Workspace Member / Workspace Admin
*   **Payload Schema** (`application/json`):
    ```json
    {
      "reason": "API Deprecation: /v1/checkout returns 503 Service Unavailable"
    }
    ```
*   **Response Schemas**:
    *   `200 OK`: Returns updated card JSON with `is_blocked = true`, `blocked_by`, `blocked_at`, and `blocked_reason`.
    *   `400 Bad Request`: Payload validation error (empty string or exceeding 255 characters limit).
    *   `403 Forbidden`: Multi-tenant tenant boundary breach or missing membership rights.
    *   `404 Not Found`: Invalid card ID or workspace mapping.

#### B. Card Unblock Toggle Endpoint
*   **Endpoint**: `POST /api/workspaces/{workspace_id}/cards/{card_id}/unblock`
*   **Access Control**: Authenticated Workspace Member / Workspace Admin
*   **Payload Schema** (`application/json`):
    ```json
    {
      "resolution": "Database lock contention fixed. Checkout microservice is stable."
    }
    ```
*   **Response Schemas**:
    *   `200 OK`: Returns updated card JSON with `is_blocked = false`, clearing out all blocker-related columns.
    *   `400 Bad Request`: Payload validation error.
    *   `403 Forbidden`: Security authorization failure.
    *   `404 Not Found`: Card or workspace not found.

#### C. Blocker Comments Retrieval API
*   **Endpoint**: `GET /api/workspaces/{workspace_id}/cards/{card_id}/block/comments`
*   **Access Control**: Authenticated Workspace Member / Workspace Viewer / Workspace Admin
*   **Response Schemas**:
    *   `200 OK`: Returns list of block comments ordered chronologically.
        ```json
        [
          {
            "id": "c62b61f8-3e4b-4b20-80a5-296531393710",
            "card_id": "893c5d6c-6a75-4c07-9b2f-376092040b2f",
            "user_id": "0d20dcfb-ef6a-4933-9118-2e06a37e1919",
            "content": "Checking DB deadlock logs on staging now.",
            "created_at": "2026-05-30T22:30:00Z",
            "updated_at": "2026-05-30T22:30:00Z"
          }
        ]
        ```

#### D. Create Blocker Comment API
*   **Endpoint**: `POST /api/workspaces/{workspace_id}/cards/{card_id}/block/comments`
*   **Access Control**: Authenticated Workspace Member / Workspace Admin
*   **Payload Schema** (`application/json`):
    ```json
    {
      "content": "deadlock query optimization PR has been opened by API team."
    }
    ```
*   **Response Schemas**:
    *   `201 Created`: Returns newly created comment JSON.
    *   `400 Bad Request`: Validation failure (content is empty or exceeds 500 characters limit).
    *   `403 Forbidden`: Unauthorized workspace access.

## 🖥️ 3. Frontend Router & Integration Specifications

The SolidJS client interface maps visual component markup, side-drawers, keyboard access, and optimistic drag-and-drop rollback physics based on standard configurations:

### 3.1 Card Blocked Component Render
*   **Path**: `apps/web/src/components/Card/Card.tsx`
*   **Visual Highlights**: Conforming to Section 12.1 of `DESIGN.md`. Renders an absolute red accent strip (`w-1 h-full bg-status-blocked absolute left-0 top-0 rounded-l-md`) inside the card container.
*   **Pulsing Rings**: Adds an active pulsing border ring to the assignee's avatar image utilizing the custom class: `ring-2 ring-status-blocked/40 animate-pulse`.

### 3.2 Blocker Details Side-Drawer
*   **Path**: `apps/web/src/components/Drawer/BlockerDrawer.tsx`
*   **Dimensions**: Slide-over panel (400px wide) emerging from the right-hand view side.
*   **A11y Focus Management & Trap**:
    1.  **Trapped Navigation Loop**: Tab key bindings loop focus elements dynamically: `Close Button` -> `Blocker Comments List` -> `Comment Textarea` -> `Submit Comment Button` -> `Resolve Block Button` -> back to `Close Button`.
    2.  **Autofocus**: The text input/textarea field receives dynamic focus immediately upon mounting.
    3.  **Escape dismissal key**: Binds a keypress listener. Pressing `Escape` closes the drawer and restores focus to the original card blocker badge element.
    4.  **Semantic Elements**: Container implements `role="dialog"`, `aria-modal="true"`, `aria-labelledby="blocker-drawer-title"`, and `aria-describedby="blocker-drawer-description"`.

### 3.3 Pragmatic Drag-and-Drop Coordinate Rollbacks & Shake
*   **Optimistic Flow**: Moving cards updates client-side layouts instantly to maintain low-latency.
*   **Network Error Catching**: Catching HTTP return errors (`409 Conflict` WIP exceeded, `422 Unprocessable Entity` Blocked card / Rule Violation) triggers rollback animations:
    1.  **Coordinate reversion**: Frontend resets the card signal array back to its original parent coordinates.
    2.  **Card Shaking**: Coordinates trigger a 300ms horizontal shake animation on the card container (`animate-shake`).
    3.  **Floating toast warning**: Renders a floating error toast warning at the bottom-right corner using the `animate-modal-pop` slide-up animation.

### 5.2 Testable Acceptance Criteria (ACs)

*   **AC5.1 (Visual Bottleneck Isolation)**: Assert cards flagged as blocked contain absolute red left border line accents, red borders, light-red translucent backgrounds, truncated shield alert text badges, and pulsating assignee avatar rings.
*   **AC5.2 (Drawer Context Slide-out)**: Clicking any card blocker badge opens right-side slide drawers showcasing full blocker descriptions, elapsed block timings, isolated comment feeds, and block-resolution actions.
*   **AC5.3 (Keyboard Navigation & Trap)**: Opening drawer captures keyboard tabs within drawer buttons. Pressing `Escape` closes the drawer and focuses the original card badge container.
*   **AC5.4 (Concurrency Toggles Safety)**: Simultaneous requests to block or unblock the same card must execute sequentially using SELECT FOR UPDATE write locks, writing exact details to transitions log tables once.
*   **AC5.5 (Multi-Tenant Scoping Safety)**: Modifying blockers or adding blocker comments must reject with HTTP `403 Forbidden` if user sessions belong to separate tenants.
*   **AC5.6 (Blocked Card Restraint)**: Dropping blocked cards inside new columns rejects immediately with `422 Unprocessable Entity` containing error code `CARD_IS_BLOCKED`.
*   **AC5.7 (Low-Latency Rollbacks & Shake)**: Rejected drag moves (Blocked cards, WIP limits, rule violations) trigger instant coordinate rollback, 300ms horizontal shake (`animate-shake`), and bottom-right alert toasts.
*   **AC5.8 (Isolated Discussions Auditing)**: Creating blocker comments adds item nodes to the card's block-discussion feed without populating the card's general comments logs. Resolving blocks clears blocker parameters but preserves discussion history for audit.
