## ⚙️ 1. Numbered Functional Requirements (FR)

### 1.1 Hierarchy Badges Rendering & Navigation (a11y)
*   **FR-70 (Parent Badge Mount)**:
    *   If a card has a non-null `parent_id`, it must render a Parent Badge on its face. The badge text must show the prefix `"↑ "` followed by the parent card's title (truncated to 20 characters with ellipses).
*   **FR-71 (Parent Navigation)**:
    *   Clicking the Parent Badge must trigger a visual focus highlight on the parent card on the Kanban board (using a 1.5s CSS flash animation `border-accent-primary animate-pulse shadow-glow`) and programmatically scroll the parent card into viewport coordinates.
*   **FR-72 (Child/Subtask Indicator)**:
    *   If a card is a parent to one or more children, it must render a Subtask Count Badge displaying `"⑆ {completed_count}/{total_count}"`.
    *   Hovering over this badge must display an accessible tooltip list (marked with `role="tooltip"`) listing the titles of all immediate subtasks and their current column status.

### 1.2 Reactive Progress Rollup
*   **FR-73 (Progress Ratio Computation)**:
    *   The frontend must compute the subtask progress percentage as `(number of children where column.is_done == true) / (total number of children) * 100`.
*   **FR-74 (Progress Bar Display)**:
    *   If the percentage is computed and `total_count > 0`, a visual progress bar (`h-1.5 w-full bg-neutral-200 dark:bg-neutral-800 rounded-full overflow-hidden mt-1.5`) must render.
    *   If `total_count == 0`, the progress bar and badges must not occupy space in the DOM.
*   **FR-75 (Real-time WebSocket Rollup Updates)**:
    *   When any card is moved to another column via WebSocket events, all ancestral cards on the board must update their progress bars dynamically within 100ms without requiring a full board refetch.

### 1.3 Decomposition Panel & Quick Add
*   **FR-76 (Drawer Panel)**:
    *   The card details drawer must contain a "Decomposition" section (`data-testid="card-decomposition-panel"`). This section lists all direct children with their completion checkbox, title, and current column name.
*   **FR-77 (Quick Subtask Creation)**:
    *   The Decomposition section must render a simple inline input field (`data-testid="decomposition-add-input"`) with a placeholder `"Add a subtask..."`.
    *   Pressing `Enter` or clicking the `"Add"` button must immediately submit a `POST /api/workspaces/:workspace_id/cards` request with the current card's ID passed as `parent_id`. The new subtask is created in the same Column and Swimlane as the parent by default.

### 1.4 Auto-Propagation Automation Rules
*   **FR-78 (In-Progress Propagation)**:
    *   If a subtask's column transitions to an active stage (where `is_done == false` and the column is not the first/backlog column), and the parent card is currently in a backlog/starting column, the backend must transition the parent card to the subtask's column (or the first in-progress column if swimlanes align). The move event transition type must be logged as `'system_auto_move'`.
*   **FR-79 (WIP Safety during Auto-Propagation)**:
    *   If the parent auto-propagation violates a target column WIP limit, the auto-propagation must abort, returning a soft notification warning to the user rather than failing the subtask's transaction.

## 💾 3. Database Integrations & Recursive DQL

The backend uses a high-performance recursive query to fetch the entire tree of a card's subtasks, fully protected against infinite cycles and excessive depth.

### 3.1 PostgreSQL Recursive CTE Query
```sql
WITH RECURSIVE hierarchy AS (
    -- Anchor member
    SELECT
        id, parent_id, workspace_id, title, current_column_id, current_swimlane_id,
        created_at, updated_at,
        1 as depth,
        ARRAY[id] as path
    FROM cards
    WHERE id = $1 AND workspace_id = $2 AND deleted_at IS NULL

    UNION ALL

    -- Recursive member
    SELECT
        c.id, c.parent_id, c.workspace_id, c.title, c.current_column_id, c.current_swimlane_id,
        c.created_at, c.updated_at,
        h.depth + 1 as depth,
        h.path || c.id as path
    FROM cards c
    INNER JOIN hierarchy h ON c.parent_id = h.id
    WHERE h.depth < 10             -- SRE: Strict max recursion depth limit (10 levels)
      AND NOT (c.id = ANY(h.path)) -- Security: Array check to prevent cyclic infinite loops
      AND c.deleted_at IS NULL     -- Exclude soft-deleted tasks
)
SELECT id, parent_id, workspace_id, title, current_column_id, current_swimlane_id, created_at, updated_at
FROM hierarchy;
```

## 🧪 4. TDD & Integration Test Blueprint

### 4.1 Rust Integration Test (`apps/api/tests/card_hierarchy_api_tests.rs`)
Validates that recursive hierarchies are retrieved correctly, cycle-protection works, and workspace-boundaries are strictly maintained under TDD:

```rust
#[sqlx::test]
async fn test_recursive_hierarchy_dql_and_cycle_boundary(pool: sqlx::PgPool) -> anyhow::Result<()> {
    sqlx::migrate!("./migrations").run(&pool).await?;

    let workspace_id = Uuid::new_v4();
    let col_id = Uuid::new_v4();
    let lane_id = Uuid::new_v4();

    // Seed base columns/swimlanes
    sqlx::query("INSERT INTO workspaces (id, name) VALUES ($1, 'Acme Portfolio')").bind(workspace_id).execute(&pool).await?;
    sqlx::query("INSERT INTO columns (id, workspace_id, title, position) VALUES ($1, $2, 'To Do', 1)").bind(col_id).bind(workspace_id).execute(&pool).await?;
    sqlx::query("INSERT INTO swimlanes (id, workspace_id, title, position) VALUES ($1, $2, 'Standard', 1)").bind(lane_id).bind(workspace_id).execute(&pool).await?;

    // 1. Create Parent Card (Level 1)
    let parent = Card::create(&pool, CreateCard {
        parent_id: None,
        workspace_id,
        title: "Parent Card".to_string(),
        current_column_id: col_id,
        current_swimlane_id: lane_id,
    }).await?;

    // 2. Create Child Card (Level 2)
    let child1 = Card::create(&pool, CreateCard {
        parent_id: Some(parent.id),
        workspace_id,
        title: "Child Card 1".to_string(),
        current_column_id: col_id,
        current_swimlane_id: lane_id,
    }).await?;

    // 3. Create Grandchild Card (Level 3)
    let grandchild = Card::create(&pool, CreateCard {
        parent_id: Some(child1.id),
        workspace_id,
        title: "Grandchild Card".to_string(),
        current_column_id: col_id,
        current_swimlane_id: lane_id,
    }).await?;

    // 4. Retrieve hierarchy from Parent (Level 1)
    let hierarchy = Card::get_hierarchy(&pool, parent.id).await?;
    assert_eq!(hierarchy.card.title, "Parent Card");
    assert_eq!(hierarchy.children.len(), 1);
    assert_eq!(hierarchy.children[0].card.title, "Child Card 1");
    assert_eq!(hierarchy.children[0].children.len(), 1);
    assert_eq!(hierarchy.children[0].children[0].card.title, "Grandchild Card");

    Ok(())
}
```

### 4.2 Playwright E2E Integration Test Blueprint (`apps/e2e/tests/card_hierarchy.spec.ts`)
Verifies reactive visual progress bar, parent badges, and quick subtask generation in the drawer:

```typescript
import { test, expect } from '@playwright/test';

test.describe('Card Recursive Hierarchy E2E', () => {
  test('should render hierarchy badges, calculate progress rollup, and support subtask creation in drawer', async ({ page }) => {
    // 1. Load Board Page
    await page.goto('/w/acme-workspace-id');

    // 2. Locate parent and child cards on the board
    const parentCard = page.locator('[data-testid^="card-container-"]').nth(0);
    const childCard = page.locator('[data-testid^="card-container-"]').nth(1);

    const parentId = await parentCard.getAttribute('data-card-id');
    const childId = await childCard.getAttribute('data-card-id');

    // Verify parent badge is visible on the child card
    const parentBadge = childCard.locator(`[data-testid="card-parent-badge-${childId}"]`);
    await expect(parentBadge).toBeVisible();
    await expect(parentBadge).toContainText('Parent');

    // Verify parent badge navigation (clicking badge highlights parent)
    await parentBadge.click();
    await expect(parentCard).toHaveClass(/animate-pulse/);

    // Verify subtask count and progress bar on parent card (e.g. ⑆ 0/1)
    const childrenBadge = parentCard.locator(`[data-testid="card-children-badge-${parentId}"]`);
    const progressBar = parentCard.locator(`[data-testid="card-progress-bar-${parentId}"]`);
    await expect(childrenBadge).toBeVisible();
    await expect(childrenBadge).toContainText('0/1');
    await expect(progressBar).toHaveAttribute('style', 'width: 0%;');

    // 3. Move the child card to 'Done' column and verify progress updates to 100%
    const doneColumn = page.locator('[data-testid="column-zone-done"]');
    await childCard.dragTo(doneColumn);

    // Assert progress bar updates reactively to 100%
    await expect(childrenBadge).toContainText('1/1');
    await expect(progressBar).toHaveAttribute('style', 'width: 100%;');

    // 4. Open parent details drawer and create a second subtask
    await parentCard.click();
    const drawer = page.locator('[data-testid="card-decomposition-panel"]');
    await expect(drawer).toBeVisible();

    const subtaskInput = drawer.locator('[data-testid="decomposition-add-input"]');
    await subtaskInput.fill('Another subtask to complete');
    await page.keyboard.press('Enter');

    // Assert new subtask appears in decomposition list and progress recalculates (now 1/2 completed = 50%)
    await expect(drawer.locator('text=Another subtask to complete')).toBeVisible();
    await expect(childrenBadge).toContainText('1/2');
    await expect(progressBar).toHaveAttribute('style', 'width: 50%;');
  });
});
```
