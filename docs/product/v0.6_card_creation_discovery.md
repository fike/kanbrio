> [!NOTE]
> This document details the strategic discovery, personas, Jobs-to-be-Done (JTBD) user stories, and interactive flow specifications for implementing the **Card Creation Flow** in Kanbrio. Building upon the 2D layout grid (v0.1), Authentication (v0.2), WIP Limits (v0.3), and Blockers Visual Language (v0.5), card creation is the vital link that allows users to actively author their workflows and dynamically test the entire board lifecycle.

---

## 👥 Part 1: Product Strategy & Rationale

Kanbrio is designed as a high-fidelity "Flow-First" management system. For a user to experience this flow, they must be able to create units of work (cards) quickly and naturally. Currently, the lack of card creation functionality locks new users out of testing our advanced features (drag-and-drop, policy interceptions, blockers discussion threads).

We evaluated two strategic paths for card creation:

### 1.1 Strategic Paths Evaluated

*   **Path A: Heavy Modal Form Dialog**
    *   *Mechanism*: Clicking "+ Add Card" opens a large central modal with extensive fields (title, type, description, assignee, checklists, priority).
    *   *Pros*: Fully comprehensive; allows users to input all metadata at the point of creation.
    *   *Cons*: Introduces massive cognitive friction; breaks the flow of editing directly on the board; feels sluggish.
*   **Path B: Lightweight Inline Card Authoring (The Selected Path)**
    *   *Mechanism*: Clicking a "+ Add Card" button at the bottom of any column/swimlane intersection collapses the button into a card-sized text area, allowing the user to type a title and hit Enter.
    *   *Pros*: High efficiency; near-zero friction; mirrors premium board products like Trello and Businessmap. Keeps the focus entirely on the board context.
    *   *Cons*: Restricts rich metadata during initial creation; requires a secondary editing drawer for advanced attributes (which will be added in subsequent cycles).

## 🧑‍🤝‍🧑 Part 2: Primary Personas

We define our card authoring strategy around three primary personas:

### 2.1 The Solo Creator ("The Swift Organizer")
*   **Core Need**: High-speed brainstorming. Wants to dump 10 tasks onto the board in under a minute without filling out forms.
*   **Pain Point**: Being forced to configure metadata fields when they only want to write down task titles.
*   **Desired Outcome**: An inline creation card where `Type -> Name -> Enter` lets them create sequential cards immediately.

### 2.2 The Project Manager ("The Structured Coordinator")
*   **Core Need**: Rules integrity. Wants to ensure that when developers create cards, they conform to active column policies (e.g. WIP limits) without failing silently.
*   **Pain Point**: Developers creating cards in columns that exceed capacity limits, causing database inconsistency or silent rule failures.
*   **Desired Outcome**: Physics-based, instant client feedback (such as a shake animation and error toast) if a card creation violates active policies.

### 2.3 The Workspace Member ("The Flow Builder")
*   **Core Need**: Visual clarity and accessibility. Needs to easily see where cards can be created, with direct focus on the active input and clear keyboard navigation.
*   **Pain Point**: Losing focus or keyboard control when triggering creation inputs, requiring mouse-clicking.
*   **Desired Outcome**: Fully accessible keyboard shortcuts (`Escape` to cancel, `Enter` to commit) that return focus to the column zone.

## 🎯 Part 3: Jobs-to-be-Done (JTBD) User Stories

### US1: Inline Card Creation on the 2D Board
*   **JTBD**: When I view the 2D board, I want to click an inline "+ Add Card" trigger at the bottom of any column and swimlane intersection, so I can quickly create a task by only typing a title and hitting Enter.
*   **Acceptance Criteria**:
    *   **AC1.1**: The bottom of each ColumnZone must render a subtle "+ Add Card" button (`text-secondary font-medium hover:text-accent-primary flex items-center gap-1.5 p-2 rounded-md hover:bg-elevated transition-colors`).
    *   **AC1.2**: Clicking "+ Add Card" must instantly collapse the button and mount an inline card-shaped input area (`p-3 bg-surface border border-base rounded-lg shadow-sm flex flex-col gap-2`).
    *   **AC1.3**: The input textarea must receive focus programmatically within 50ms.
    *   **AC1.4**: Pressing `Enter` must submit the card creation, and pressing `Escape` or clicking outside must cancel and restore the "+ Add Card" button.

### US2: Validation, Errors, & Policy Enforcement
*   **JTBD**: When I submit a new card, I want the system to validate the input and enforce WIP limit policies, so that the board's integrity is maintained and I receive immediate visual feedback on errors.
*   **Acceptance Criteria**:
    *   **AC2.1**: Input must be non-empty, stripped of leading/trailing whitespace, and contain a maximum of 255 characters.
    *   **AC2.2**: The backend must check WIP limits on the target Column/Swimlane. If creation violates a WIP limit, the backend returns `409 Conflict`, the inline card shakes (`animate-shake`), and a red error toast details the violation.
    *   **AC2.3**: During API submission, the inline input and action buttons must enter a disabled, `aria-disabled="true"` loading state, showing a spinner.

### US3: Multi-tenant Security Boundaries
*   **JTBD**: When a card is created, I want the system to ensure it is isolated within my workspace boundary, so that other tenants cannot access or tamper with our private tasks.
*   **Acceptance Criteria**:
    *   **AC3.1**: The client POST request must target `/api/workspaces/:workspace_id/cards` and pass through `WorkspaceTenantGuard`.
    *   **AC3.2**: Creation must execute inside a database transaction, logging a `'create'` audit event to the `card_transitions` table.

## 🎨 Part 4: Interactive & UX Specifications

### 4.1 Inline Component Transitions
*   **Button-to-Input**: Replaced with a smooth 150ms crossfade.
*   **Active Authoring Card**: Size matches standard cards. Implements `bg-surface border-base shadow-sm rounded-lg`.
*   **Action Row**: Below the text area, renders two action buttons: `Add` (`bg-accent-primary text-white font-semibold py-1 px-3 rounded-md hover:bg-accent-primary/95 text-xs`) and `Cancel` (`text-secondary font-semibold hover:bg-elevated px-3 py-1 rounded-md text-xs`).

### 4.2 Playwright Test Anchors (`data-testid`)
*   Add Card Button: `data-testid="column-add-card-button-{column_id}-{swimlane_id}"`
*   Inline Card Form: `data-testid="inline-card-form-{column_id}-{swimlane_id}"`
*   Card Title Input: `data-testid="inline-card-title-input"`
*   Add Card Submit: `data-testid="inline-card-submit"`
*   Add Card Cancel: `data-testid="inline-card-cancel"`
*   Error Message Banner: `data-testid="inline-card-error"`
