> [!NOTE]
> This document details the strategic discovery, personas, Jobs-to-be-Done (JTBD) user stories, and interactive flow specifications for implementing the **Recursive Task Hierarchy & Visualization** in Kanbrio. Building upon the 2D layout grid (v0.1), Authentication (v0.2), WIP Limits (v0.3), and Card Component (v0.6), task hierarchy allows users to decompose complex deliverables into recursive parent-child relationships and track execution progress directly on the board.

---

## 👥 Part 1: Product Strategy & Rationale

As Kanbrio scales to support larger teams and complex projects, a flat 2D board of cards becomes insufficient. Features, initiatives, and user stories naturally decompose into smaller technical subtasks. However, traditional Kanban systems either force users to navigate away from the board to see child tasks, or they clutter the interface with massive, rigid tree views that destroy board responsiveness.

Our strategic objective is to introduce **Recursive Task Hierarchy & Visualization (v0.7)** directly on the 2D Kanban board in a way that is visual, low-friction, and high-performance.

We evaluated three strategic paths for hierarchy visualization:

### 1.1 Strategic Paths Evaluated

*   **Path A: Separate Portfolio Tree View**
    *   *Mechanism*: A dedicated hierarchical directory or tree diagram page separate from the Kanban board.
    *   *Pros*: Uncluttered; provides infinite depth visualization without space constraints.
    *   *Cons*: High context-switching cost; developers lose board-level visibility of current work; hard to manage drag-and-drop.
*   **Path B: Nested Cards directly inside Cards (Russian Doll style)**
    *   *Mechanism*: Cards physically contain nested sub-cards on the Kanban board grid.
    *   *Pros*: Incredibly visual; immediate relation visible.
    *   *Cons*: Causes massive board layout distortion; ruins column vertical spacing; severely impacts rendering performance when nesting exceeds 2 levels.
*   **Path C: Hybrid Card Indicators with Progress Rollup & Quick Tree Modal (The Selected Path)**
    *   *Mechanism*: Parent cards remain normal size but render:
        1. A reactive subtask progress bar (showing completed vs total subtasks).
        2. A hierarchy badge indicating parent/child links.
        3. Clickable hierarchy visual breadcrumbs/drawers.
        4. Inline subtask decomposition list on card expand.
    *   *Pros*: Perfect balance of visual clarity and board density. Maintains low-latency Solid.js performance while offering immediate progress visibility and click-to-traverse controls.
    *   *Cons*: Requires a secondary drawer or tree viewer modal to inspect the entire multi-level recursive path.

## 🧑‍🤝‍🧑 Part 2: Primary Personas

We define our hierarchy visualization strategy around three primary personas:

### 2.1 The Swift Task Decomposer ("The Focused Builder")
*   **Core Need**: High-speed breakdown of larger initiatives. Wants to create subtasks inline inside a parent card drawer without leaving the board or reloading pages.
*   **Pain Point**: Being forced to navigate to a "new task page" and manually copy-paste fields to link a subtask to its parent.
*   **Desired Outcome**: A rapid subtask creator (`Title -> Enter`) inside the card view that automatically sets `parent_id` and registers the child instantly.

### 2.2 The Delivery Manager ("The Progress Tracker")
*   **Core Need**: Status transparency. Needs to know the exact progress of complex cards at a single glance without micro-managing team members or clicking into individual cards.
*   **Pain Point**: Subtasks being updated but the parent card remaining visually static, forcing manual status tracking.
*   **Desired Outcome**: A visual rollup progress bar on parent cards showing the exact completion percentage of its subtasks, which updates in real-time as subtasks move to 'Done'.

### 2.3 The Workspace Coordinator ("The Flow Optimizer")
*   **Core Need**: Visual context and navigation. Needs to trace card dependencies and hierarchy relationships (where does this subtask belong?) immediately from any column.
*   **Pain Point**: Navigating a cluttered board where orphan subtasks are mixed with parent initiatives, making it hard to see the big picture.
*   **Desired Outcome**: Clear breadcrumbs and visual badges on cards showing the parent relationship, with click-to-locate actions to highlight relative cards.

## 🎯 Part 3: Jobs-to-be-Done (JTBD) User Stories

### US1: Visual Parent-Child Indicators & Navigation
*   **JTBD**: When I look at a card on the board, I want to see clear, high-contrast badges representing its hierarchical relationships (i.e., whether it has a parent or children), so that I can quickly navigate between related cards without losing my context.
*   **Acceptance Criteria**:
    *   **AC1.1**: If a card has a parent, the card body must render a "Parent Badge" (`bg-elevated text-secondary border border-base flex items-center gap-1 py-0.5 px-1.5 rounded text-[10px] w-fit font-medium hover:text-accent-primary transition-colors cursor-pointer`) containing the parent's short title.
    *   **AC1.2**: Clicking the Parent Badge must visually highlight the parent card on the board (e.g., flash a green border `border-accent-primary shadow-glow` for 1.5s) and scroll it into view.
    *   **AC1.3**: If a card has children, it must render a "Subtask Badge" showing the count of children (e.g., `⑆ 3 Subtasks`).
    *   **AC1.4**: Hovering over the Subtask Badge must show a tooltip list of child titles and their status.

### US2: Subtask Progress Rollup Bar
*   **JTBD**: When a card has subtasks, I want to see a progress bar indicating the percentage of completed subtasks directly on the card face, so that I can track work completion at a glance.
*   **Acceptance Criteria**:
    *   **AC2.1**: A card with child tasks must render a subtle horizontal progress bar (`h-1.5 bg-neutral-200 dark:bg-neutral-800 rounded-full overflow-hidden mt-1.5 w-full`) above the card footer.
    *   **AC2.2**: The progress fill bar must use theme-appropriate styling (`bg-accent-primary h-full transition-all duration-300`) and calculate the completion ratio as `(Completed Children Count) / (Total Children Count) * 100%`.
    *   **AC2.3**: If a card has 0 children, the progress bar must not render, avoiding unnecessary visual clutter.
    *   **AC2.4**: Completed children are defined as child cards whose column has `is_done = true`.

### US3: Quick Decomposition & Parent Sync Rules
*   **JTBD**: When I manage a parent card, I want to quickly add subtasks or have the parent card's status react to its subtasks' movement, so that the workflow is automated and manual updates are minimized.
*   **Acceptance Criteria**:
    *   **AC3.1**: Inside a card's details drawer, a "Decomposition Panel" must list all active child tasks, showing their current column, assignee, and a quick "+ Add Subtask" text field.
    *   **AC3.2**: When a subtask moves from an in-progress column to a done column, the parent card's progress bar must animate to reflect the change in real-time (via WebSocket sync).
    *   **AC3.3**: *Automation Rule (State Propagation)*: When the first child card moves from `Backlog/To Do` into `In Progress`, if the parent card is still in `To Do`, the parent card should automatically transition to the `In Progress` column (and log the transition as `system_auto_move`).
    *   **AC3.4**: *Automation Rule (Completion Propagation)*: When the last remaining incomplete child card moves to a Done column, the system should prompt the user (or automatically if configured) to move the parent card to the Done column.

## 🎨 Part 4: Interactive & UX Specifications

### 4.1 Card Hierarchy UI Components
*   **Parent Badge**: Positioned at the top-left of the card content, below any type tags.
*   **Subtask Progress Bar**: Positioned at the bottom of the card, spanning the full width, just above the assignee avatar and card code identifier.
*   **Hierarchy Context Modal**: Right-clicking a card or clicking a details icon opens a visual hierarchy tree overlay using a fast Canvas/Mermaid-like micro-graph to show the recursive ancestors and descendants up to 5 levels deep.
