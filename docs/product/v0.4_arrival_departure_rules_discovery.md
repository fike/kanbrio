This feature builds directly upon the 2D Board context (v0.1) and Multi-Tenant Workspace boundaries (v0.2), ensuring that cards transitioning between columns and swimlanes adhere to strict qualitative gates to maximize Lead Time predictability and minimize waste.

## 👥 Part 1: Product Discovery & Requirements

In Lean manufacturing and high-maturity Kanban (inspired by Businessmap/Kanbanize), teams prevent defects and "unripe" work from moving forward by defining explicit **Policies** (or Rules) on columns. Kanbrio introduces these as automated gates at the edge of board transitions.

### 1.1 Primary Personas

#### 1.1.1 The Agile Coach (The "Flow Guardian")
*   **Core Need**: Enforce explicit policies (e.g. "Do not pull into 'Testing' without a Test Plan checklist completed" or "Do not move from 'Development' without all subtasks resolved").
*   **Pain Point**: Standard columns relying on human memory to double-check checklist items or fields, leading to incomplete work moving forward (defects) and high rework rates.
*   **Desired Outcome**: Automated validation of column transit rules that block invalid moves and display clear explanations of what policy is violated.

#### 1.1.2 The Individual Developer (The "Focused Executor")
*   **Core Need**: Direct, contextual checklist items directly on the card to guide their execution path, and instant feedback if they attempt an invalid drag-and-drop.
*   **Pain Point**: Vague descriptions, forgotten steps, and frustration when cards are kicked back from QA because a small verification task was forgotten.
*   **Desired Outcome**: Interactive, highly responsive checkbox checklists on the card footer and a shake animation explaining missing criteria on invalid drop.

#### 1.1.3 The Workspace Admin (The "Policy Architect")
*   **Core Need**: Easily manage policies per column and swimlane, define custom checklist requirements, and override rule blocks for critical production hotfixes.
*   **Pain Point**: Complex, rigid configuration engines.
*   **Desired Outcome**: Simple sidebar rule configurations directly scoping active column boundaries with clear override toggles.

### 1.2 Jobs-to-be-Done (JTBD) User Stories

#### US1: Managing Card Checklist Items
*   **JTBD**: When I am working on a complex card, I want to create, reorder, check off, and delete checklist items directly on the card, so that I can break down my execution steps and track granular progress.
*   **Acceptance Criteria**:
    *   **AC1.1**: Any member can CRUD (Create, Read, Update, Delete) checklist items on cards belonging to their workspace.
    *   **AC1.2**: Toggling a checklist item updates progress indicator metrics (e.g. `[Checklist: 2/5]`) in real-time.
    *   **AC1.3**: Checked items record the completing user's ID and timestamp for event auditing.

#### US2: Configuring Column-Specific Transition Rules
*   **JTBD**: When I am establishing workspace governance policies, I want to bind specific validation rules (e.g. "Assignee required on entry", "Checklists completed on exit") to board columns, so that we can systematically prevent invalid transitions.
*   **Acceptance Criteria**:
    *   **AC2.1**: Only Workspace Admins can edit or delete transition rules. Members receive a `403 Forbidden` response.
    *   **AC2.2**: The rules must be scoped strictly to a column and distinguish between **Arrival Rules** (entry) and **Departure Rules** (exit).
    *   **AC2.3**: Supported rule criteria:
        1.  `assignee_required`: Card must have a valid assignee.
        2.  `checklist_completed`: All checklist items on the card must be completed.
        3.  `subtasks_completed`: All hierarchical child subtasks must be in a 'Done' column.

#### US3: Drag & Drop Validation and Reject Shaking
*   **JTBD**: When I drag a card to a column whose entry policies are not satisfied, I want the system to reject the move, return the card to its starting column, and display a highly visible explanation of the unmet policy, so I can instantly correct it.
*   **Acceptance Criteria**:
    *   **AC3.1**: Moving an invalid card must trigger an optimistic UI rollback, return the card to its original coordinate, and trigger a 300ms horizontal shake animation (`animate-shake`).
    *   **AC3.2**: Renders a custom Red Alert Toast detailing the policy mismatch (e.g. `[Arrival Rule Violation]: QA requires an Assignee before entry.`).

### 1.3 Numbered Functional Requirements (FR)

*   **FR25**: Cards must support an independent list of interactive `checklist_items` mapping `id`, `card_id`, `title`, `is_completed`, and `position`.
*   **FR26**: The database must enforce a `transition_rules` relation scoping workspace column boundaries by active policy parameters (`rule_type` as entry/exit, `criteria_type`).
*   **FR27**: Backend card movement transactions must intercept PUT/PATCH/MOVE endpoints, query column transition rules for both the source (departure) and target (arrival) columns, and reject the transaction on policy violations with HTTP `422 Unprocessable Entity` and a specific error payload.
*   **FR28**: Admins must have an explicit `override` option to bypass rules, writing a dedicated transition event type `[move_rule_override]` to the immutable transition log.

## 🎨 Part 2: UX & Interaction Specifications

### 2.1 Micro-Animations & Board Feedback
*   **Checklist Checkbox Toggle**: When clicking a checkbox, trigger a smooth scale-up pop (`scale-110`) on the checkbox asset and cross-out the text using `text-tertiary line-through` with standard duration (`duration-150 ease-standard`).
*   **Invalid Drop Shake**: Same high-quality animation as WIP limit violations. The dropped card shakes horizontally, signaling a physics rollback, while the top-edge shimmer bar briefly flashes red.

### 2.2 Playwright Test Anchors (`data-testid`)
*   Checklist Container: `data-testid="card-checklist-container"`
*   Checklist Item: `data-testid="checklist-item-{id}"`
*   Checkbox: `data-testid="checklist-checkbox-{id}"`
*   Add Checklist Input: `data-testid="add-checklist-input"`
*   Rule Badge on Column Header: `data-testid="column-rule-indicator-{column_id}"`
*   Rule Violation Toast: `data-testid="rule-violation-toast"`
