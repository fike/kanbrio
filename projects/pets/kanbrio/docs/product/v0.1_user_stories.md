# v0.1 Foundation: User Stories

> [!TIP]
> This document defines the *Why* and *Who*. For the technical *How*, see the [v0.1 Mini-PRD](v0.1_mini_prd.md).

These user stories focus on the *outcomes* and *motivations* of our primary personas for the first release.

## 👥 Persona: The Project Manager (Predictability)

### US1: Recursive Task Structure
**As a** Project Manager,
**I want** to create a parent initiative and link multiple child tasks to it,
**So that** I can see how granular work contributes to the high-level roadmap without manual tracking.
*   **Acceptance Criteria**:
    *   System supports infinite nesting (Parent -> Child -> Grandchild).
    *   Deleting a parent does not orphan children (handled via soft-delete or re-parenting).

### US2: 2D Context (Columns x Swimlanes)
**As a** Project Manager,
**I want** to see my tasks organized by 'Status' (columns) and 'Category' (swimlanes like Expedite),
**So that** I can prioritize urgent work without losing sight of the standard flow.
*   **Acceptance Criteria**:
    *   Board displays horizontal rows (swimlanes).
    *   A card must always belong to one Column AND one Swimlane.

## 👥 Persona: The Agile Coach (Flow Optimization)

### US3: Transition Auditing
**As an** Agile Coach,
**I want** the system to record the exact timestamp when a card moves between columns,
**So that** I can later calculate Lead Time and identify bottlenecks.
*   **Acceptance Criteria**:
    *   Every movement generates an immutable record in `card_events`.
    *   Record includes: `card_id`, `from_col`, `to_col`, `timestamp`.

### US4: Multi-level WIP Limits
**As an** Agile Coach,
**I want** to set a maximum card limit for a specific column,
**So that** I can prevent the team from starting too much work and slowing down the overall flow.
*   **Acceptance Criteria**:
    *   Backend blocks (or warns) movement if the column count >= WIP Limit.
