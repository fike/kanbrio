# Mini-PRD: v0.1 The Skeleton

**Status**: Draft | **Version**: 0.1 | **Owner**: @product-manager

> [!IMPORTANT]
> These technical requirements implement the value defined in the [v0.1 User Stories](v0.1_user_stories.md).

## 1. Problem Statement
Users cannot manage complex hierarchical project portfolios in current open-source tools without experiencing "data silos" or losing the ability to track granular flow metrics (timestamps).

## 2. Objective
Establish the **Data Core** and **Basic Visualization** for a 2D Kanban board with native recursive hierarchy and event logging.

## 3. Functional Requirements

### 3.1 Recursive Node Architecture
- **FR1**: The system must use a UUID-based adjacency list to support infinite Parent-Child relationships.
- **FR2**: API must provide a way to fetch a "Subtree" of tasks for a given initiative.
- **FR3**: Status roll-up logic (e.g., Initiative % complete) must be calculated on-the-fly using Recursive CTEs.

### 3.2 2D Workflow Engine
- **FR4**: Boards must support configurable **Vertical Columns** and **Horizontal Swimlanes**.
- **FR5**: Cards must possess a `column_id` and a `swimlane_id`.
- **FR6**: The system must support **WIP Limits** at the Column and Swimlane levels.

### 3.3 Event Sourcing (Audit Log)
- **FR7**: Every state change (create, move, block, edit) must trigger an immutable event entry.
- **FR8**: Event schema: `id`, `card_id`, `actor_id`, `event_type`, `payload (jsonb)`, `created_at`.

## 4. Technical Constraints
- **Database**: PostgreSQL 14+ (Required for performance optimizations of Recursive CTEs).
- **Integrity**: Card moves and Event creation must occur within a single database transaction.

## 5. Success Metrics
- **Performance**: Fetching a 5-level deep hierarchy subtree in < 100ms.
- **Integrity**: 100% of status changes must have a corresponding event log entry.
